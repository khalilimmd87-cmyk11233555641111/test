# 🚀 RVG Gateway – تیم آزادی

دروازه (Gateway) سریع و مدرن برای تونل‌زنی VLESS روی WebSocket و XHTTP، با داشبورد مدیریتی و قابلیت ساخت لینک‌های اختصاصی با محدودیت ترافیک.

لینک ویدیو اموزش
https://youtu.be/nTq2vDmC0Ks

---

## ✨ ویژگی‌ها

- 🔌 تونل VLESS روی WebSocket (TLS 443)
- ⚡ تونل VLESS روی XHTTP (دو مد `packet-up` و `stream-up`، با موتور تطبیقی برای throughput بالا)
- 🌐 HTTP Proxy داخلی (با محافظت SSRF در برابر آدرس‌های داخلی/متادیتا)
- 📊 داشبورد مدیریتی کامل (آمار، نمودار ترافیک، اتصالات زنده)
- 🔗 مدیریت لینک‌های نامحدود با محدودیت ترافیک اختصاصی (MB/GB) و گروه‌بندی (Sub Groups)
- ✅ فعال/غیرفعال‌سازی هر لینک به‌صورت لحظه‌ای
- 🛡️ Fingerprint Spoofing (Chrome) + محافظت پایه در برابر active-probing

## 2️⃣ Deploy روی Railway

1. وارد [Railway.app](https://railway.app) شوید.
2. **New Project → Deploy from GitHub repo** → ریپازیتوری خود را انتخاب کنید.
3. یک **Volume** با mount path `/data` به سرویس اضافه کنید (بخش «پرسیستنس» زیر را بخوانید — بدون این کار، رمز عبور و لینک‌ها با هر ریستارت پاک می‌شوند).
4. متغیرهای محیطی زیر را تنظیم کنید (بخش بعد).
5. یک دامنه‌ی عمومی (Public Domain) برای سرویس فعال کنید تا `RAILWAY_PUBLIC_DOMAIN` درست مقداردهی شود.

## ⚙️ متغیرهای محیطی

| متغیر | توضیح | پیش‌فرض |
|---|---|---|
| `ADMIN_PASSWORD` | رمز اولیه‌ی ورود به پنل. **حتماً بعد از اولین دیپلوی از پنل تغییرش بدهید.** | `TimAzadi` |
| `SECRET_KEY` | کلید امنیتی برای هش/سشن. اگر ست نشود، خودکار ساخته و روی دیسک (`/data/.secret_key`) ذخیره می‌شود. | تصادفی |
| `DATA_DIR` | مسیر ذخیره‌ی state دائمی. باید روی یک Railway Volume باشد. | `/data` |
| `RAILWAY_PUBLIC_DOMAIN` | دامنه‌ی عمومی سرویس؛ Railway خودش ست می‌کند. | — |
| `ALLOWED_ORIGINS` | لیست (با کاما جدا) دامنه‌های اضافی مجاز برای CORS، اگر پنل را از دامنه‌ی دیگری هم صدا می‌زنید. | — |
| `DISABLE_SECURE_COOKIE` | فقط برای اجرای لوکال روی `http://localhost` (بدون TLS) به `1` ست کنید. **در پروداکشن هرگز.** | `0` |

## ⚠️ پرسیستنس (خیلی مهم)

State (لینک‌ها، آمار مصرف، رمز عبور، secret) روی فایل در `DATA_DIR` (پیش‌فرض `/data`) ذخیره می‌شود. اگر روی Railway یک **Volume** به همین مسیر متصل نکرده باشید، فایل‌سیستم موقتی است و با هر ریستارت/دیپلوی:
- رمز عبور به مقدار پیش‌فرض (`ADMIN_PASSWORD`) برمی‌گردد،
- تمام لینک‌ها و آمار مصرف پاک می‌شوند،
- `SECRET_KEY` جدید ساخته می‌شود و سشن‌های فعال باطل می‌شوند.

برای پرسیستنسی واقعی، حتماً از Railway Volume (یا در آینده Redis/PostgreSQL) استفاده کنید.

## 🔐 نکات امنیتی

- بلافاصله بعد از اولین ورود، رمز عبور را از پنل تغییر بدهید.
- `ALLOWED_ORIGINS`/CORS فقط به دامنه‌ی واقعی سرویس محدود شده؛ اگر پنل را از جای دیگری (مثلاً یک فرانت‌اند جدا) صدا می‌زنید، دامنه‌اش را به `ALLOWED_ORIGINS` اضافه کنید — هرگز به wildcard برنگردانید چون با `credentials` همراه است.
- تلاش‌های ورود ناموفق rate-limit شده‌اند (۵ تلاش هر ۵ دقیقه به‌ازای هر IP).
- مسیر `/proxy/...` فقط برای ادمین لاگین‌شده در دسترس است و به آدرس‌های داخلی/متادیتای کلود وصل نمی‌شود.

## 📡 درباره‌ی پروتکل‌ها و ضد-فیلترینگ

این سرویس روی **Railway با uvicorn** اجرا می‌شود، یعنی TLS توسط لایه‌ی Railway/Cloudflare (نه خود اپ) ترمینیت می‌شود و هیچ پورت TCP/UDP خامی مستقیماً exposed نیست. این محدودیت زیرساختی تعیین می‌کند چه چیزی واقعاً قابل‌اجراست:

- ✅ **VLESS over WebSocket** و **VLESS over XHTTP** (پکت‌آپ و استریم‌آپ با موتور تطبیقی AIMD) — همان چیزی که الان پیاده‌سازی شده و در حال حاضر یکی از به‌روزترین روش‌های عبور از DPI مبتنی بر HTTP هستند.
- ❌ **REALITY** نیاز به کنترل کامل TLS handshake در لایه‌ی TCP خام دارد؛ چون TLS اینجا قبل از رسیدن به اپ توسط Railway ترمینیت می‌شود، پیاده‌سازی واقعی REALITY روی این معماری ممکن نیست.
- ❌ **Hysteria2** روی QUIC/UDP کار می‌کند؛ Railway پورت UDP خام برای اپ‌های معمولی expose نمی‌کند.
- ❌ **gRPC transport واقعی** به HTTP/2 سمت سرور نیاز دارد؛ uvicorn برای درخواست‌های ورودی HTTP/2 را serve نمی‌کند (فقط برای خروجی از httpx). بدون تغییر کامل استک به یک سرور HTTP/2 (مثل Hypercorn) و بدون امکان تست واقعی با کلاینت‌های v2rayNG/NekoBox در این محیط، افزودنش را غیرقابل‌اعتماد و بالقوه گمراه‌کننده تشخیص دادم و انجامش ندادم.

برای همین، به‌جای افزودن پروتکل «جدید» غیرقابل‌تست، تقویت‌های واقعی زیر روی همان WS/XHTTP موجود اعمال شد:
- حذف پیام‌های خطای اختصاصی (مثل `"not authorized"`) که امضای این کدبیس بودند، و جایگزینی با پاسخ‌های عمومی‌تر (404 ساده) در برابر active-probing.

اگر بعداً به یک VPS با دسترسی کامل به TCP/UDP خام مهاجرت کردید (به‌جای Railway)، REALITY و Hysteria2 گزینه‌های واقعاً قابل‌اجرا می‌شوند و می‌توانیم آن‌ها را اضافه کنیم.
